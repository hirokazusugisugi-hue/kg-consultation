# 要件定義書：担当者自動選定システム

**作成日**: 2026-04-01
**対象**: KG中小企業経営診断研究会 無料経営相談システム

---

## 1. 背景・目的

現行システムでは、日程ごとにメンバーが参加表明し、表明者全員がそのまま確定通知の対象となる。
この方式では特定メンバーに参加が偏る問題がある。
確定時にシステムが全アクティブメンバーから自動選定することで、参加の均等化と適正人数の確保を実現する。

## 2. 現行仕様

| 項目 | 現行 |
|------|------|
| メンバー選定方式 | 日程ごとに手動参加表明（日程設定シートH列） |
| 担当者人数 | 制限なし（参加表明者全員） |
| 配置点数 | 1期/2期診断士=2点、3期/4期オブザーバー=1点 |
| 予約可能判定 | 診断士1名以上 かつ 4点以上（or 2点以上+特別対応） |
| 確定フロー | 参加表明 → 会場選択 → 確定 → 全参加者に通知 |

## 3. 変更要件

### 3.1 根本方針の変更

**日程ごとの事前参加表明を廃止し、確定時に全アクティブメンバーから自動選定する。**

| 項目 | 現行 | 変更後 |
|------|------|--------|
| 選定元 | 日程に参加表明したメンバー | **全アクティブメンバー**（顧問除く） |
| 選定タイミング | なし（全員参加） | **確定時**（setVenueAndConfirm） |
| 選定方法 | なし | **テーママッチ優先 + ランダム** |

### 3.2 担当者人数制約

| 項目 | 通常時 | 特別対応時 |
|------|--------|-----------|
| 選出人数 | **4名** | 2名以上 |
| 確定人数 | **3名** | 2名以上 |
| 予備人数 | **1名** | 0名〜 |
| 必須診断士数 | 1名以上 | 1名以上 |
| 配置点数閾値 | 4点以上 | 2点以上 |

### 3.3 確定フローの変更

```
【現行】
メンバーが日程に参加表明 → 会場選択 → 確定 → 全参加者に通知

【変更後】
会場選択 → 確定
  → システムが全メンバーから4名を自動選定
  → 4名から3名確定 + 1名予備
  → 確定3名のみに通知
```

### 3.4 自動選定ロジック

#### ステップ1: 候補プールの構築

メンバーマスタから以下の条件で候補を取得：
- `active = TRUE`（システム参加）
- `type ≠ '顧問'`（顧問は対象外）

候補を2グループに分類：
- **診断士グループ**: 1期/2期（1人あたり2点）
- **OBグループ**: 3期/4期オブザーバー（1人あたり1点）

#### ステップ2: 診断士の選定（テーママッチ優先）

1. 相談申込の「相談テーマ」（I列）を取得
2. 診断士グループの各メンバーの「得意テーマ」（メンバーマスタJ列）とマッチング
3. **テーマが一致する診断士を優先選出**（1名以上）
4. テーマ一致者が複数いる場合はランダムに1名
5. テーマ一致者がいない場合は診断士からランダムに1名

#### ステップ3: 残り枠の充填（ランダム）

1. ステップ2で選ばれなかった全メンバー（診断士+OB）をランダムシャッフル
2. 合計4名になるまで順に追加
3. 追加後、配置点数が4点以上か検証
4. 点数不足の場合: 低得点メンバーを未選出の診断士と入れ替え

#### ステップ4: 3名確定 + 1名予備

4名から1名を予備メンバーとする：
1. OBを優先的に予備候補とする
2. 以下の制約を満たす範囲でランダムに予備を決定:
   - 残り3名に診断士が1名以上
   - 残り3名の配置点数が4点以上
3. 制約を満たせない場合、4名全員を確定とする

#### ステップ5: 最終バリデーション

**通常時** — 以下をすべて満たすこと：
- 診断士（1期/2期）が1名以上
- 配置点数が4点以上
- 確定メンバーが3名（制約上やむを得ない場合は4名）

**特別対応時** — 以下をすべて満たすこと：
- 診断士（1期/2期）が1名以上
- 配置点数が2点以上
- 確定メンバーが2名以上

**失敗時**: 確定を中止、エラーメッセージを返す

### 3.5 テーママッチングの仕様

メンバーマスタの「得意テーマ」列（カンマ区切り）と、申込の「相談テーマ」を比較。

| 得意テーマ（例） | 相談テーマ（例） | マッチ |
|----------------|----------------|--------|
| 経営戦略,マーケティング | マーケティング | YES |
| 財務,IT活用 | 経営戦略 | NO |
| 経営企画,DX,生成AI | DX推進 | YES（部分一致） |

- カンマ区切りの各テーマについて、相談テーマとの部分一致で判定
- 大文字小文字は区別しない

### 3.6 予備メンバーの扱い

| 項目 | 内容 |
|------|------|
| 記録場所 | 予約管理シートの備考列（T列）に「予備: ○○」と記録 |
| 通知 | 確定通知は送信しない |
| 欠員時の繰り上げ | 確定メンバーに欠員が出た場合、予備メンバーに依頼 |
| 繰り上げ方法 | 手動（ポータルからP列を編集し再通知） |

## 4. 影響範囲

### 4.1 設定ファイル（config.gs）

```javascript
STAFF_LIMITS: {
  SELECT: 4,               // 第一段階の選出人数
  FINAL: 3,                // 最終確定人数
  MIN_SPECIAL: 2,          // 特別対応時の最小人数
  REQUIRE_SHINDANSHI: 1    // 必須診断士数（1期/2期）
}
```

### 4.2 メンバー管理（members.gs）

新規関数：
- `selectStaffMembers(consultTheme, specialFlag)`
  - 入力: 相談テーマ（文字列）、特別対応フラグ
  - 出力: `{ success, confirmed: [メンバー名], reserve: [メンバー名], score, message }`
  - 処理: ステップ1〜5のロジック
- `matchTheme(memberThemes, consultTheme)`
  - 入力: メンバーの得意テーマ（カンマ区切り）、相談テーマ
  - 出力: boolean（マッチするかどうか）

### 4.3 ポータル確定処理（portal.gs）

`setVenueAndConfirm()` を改修:
- 確定前に `selectStaffMembers()` を呼び出し
- 確定メンバーをP列（担当者）に書き込み
- 予備メンバーをT列（備考）に追記
- **確定メンバーのみ**に確定通知を送信

### 4.4 日程設定シート（既存のシフト参加機能）

- 日程設定シートH列の参加表明機能は**残す**（予約可能判定のため引き続き使用）
- ただし確定時の担当者選定には使用しない（全メンバーから選定）
- ポータルのシフト参加UI自体は削除不要（日程の出欠確認用途として存続可能）

### 4.5 予約可能判定（members.gs）

- `getBookableStatus()` は変更なし（閾値4点据え置き）
- 予約可能判定は引き続き日程設定シートの参加メンバーで判定

### 4.6 ポータルUI（portal.html）

- 確定結果に「確定: ○○、△△、□□（3名/6点）予備: ◇◇」を表示

## 5. 非機能要件

- 選定結果はP列（担当者）に記録される
- テーママッチ＋ランダム選定のため、同一条件でも結果が変わりうる
- 手動でP列を編集済みの場合はそちらを優先（自動選定スキップ）

## 6. 将来拡張

- 確定人数を3名→2名に変更（CONFIG変更のみ）
- 参加回数の均等化（履歴を参照して少ない人を優先）
- 得意業種マッチング追加（テーマに加え業種でもマッチ）
