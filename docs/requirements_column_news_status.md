# メディアサイト構築 + ステータスフロー改定 要件定義

---

## 全体構想

### 背景

現在、お知らせ・コラム（記事）はGASの管理画面から個別に管理しているが、
**閲覧者向けの独立したメディアサイト**が存在しない（`site/articles.html` は「準備中」のプレースホルダー）。

LPに直接コンテンツ機能を組み込むと構造が崩れるリスクがあるため、
**まずメディアサイトを独立して構築**し、将来的にLPとの相互リンク・遷移で連携する。

### 方針

```
【現在のシステム構成】

LP（index15.html）        ←→  GAS API  ←→  スプレッドシート
PRサイト（site/index.html）←→  GAS API
ポータル（site/portal.html）←→  GAS API

【今回追加するもの】

メディアサイト（site/media/）←→  GAS API  ←→  記事管理シート
                             ←→  さくらPHP（画像・音声）
                             ←→  note.com RSS（外部連携）
```

### 段階的な連携イメージ

| Phase | 内容 | LP・ポータルとの関係 |
|-------|------|---------------------|
| **Phase 1** | メディアサイト単独構築（ニュース・コラム・Podcast閲覧） | **独立運用** — ナビにリンクを追加するのみ |
| **Phase 2** | LP・PRサイトのナビゲーションにメディアサイトへの遷移導線を追加 | **リンク連携** — LP ↔ メディアサイト間の相互遷移 |
| **Phase 3** | LPのニュースティッカーにメディアサイトの最新記事を表示 | **データ連携** — 同じGAS APIからデータ取得 |

---

## 変更点サマリ

| 項目 | 現状 | 変更後 |
|------|------|--------|
| メディアサイト | なし（「準備中」ページのみ） | **独立メディアサイト `site/media/` を新規構築** |
| お知らせ表示 | LPに直接表示 | **メディアサイトで一覧表示 + LP連携（Phase 3）** |
| コラム表示 | なし | **メディアサイトで一覧・詳細表示** |
| コラム写真 | サムネイルURL 1枚のみ | **本文中に複数画像挿入可能** |
| Podcast | 未対応 | **音声ファイルアップロード + 再生 + QRコード** |
| note.com連携 | なし | **RSSフィード取込で自動掲載（サーバーサイド）** |
| ニュース自動投稿 | 手動のみ | **コラム公開時にニュースへ自動投稿** |
| ステータス「確定」 | `確定` | **`予約確定`** |
| ステータス「完了」 | アンケート送付で完了 | **アンケート受領で完了** |
| 最終完了 | なし | **報告書送付で「最終完了」新設** |

### 1-3. デザインシステム

既存PRサイト（`site/css/style.css`）のデザイン変数を継承:

| 項目 | 値 |
|------|-----|
| プライマリカラー | `--kgu-blue: #0F2350` |
| アクセントカラー | `--kgu-gold: #C5A15A` |
| フォント | `Noto Sans JP` / `Noto Serif JP` |
| 最大幅 | `1200px`（既存準拠） |
| ヘッダー・フッター | PRサイト共通コンポーネントを使用 |

`media.css` では記事カード、Podcastプレイヤー、ページネーション等のメディアサイト固有スタイルのみ追加。

### 1-4. データ取得フロー

```
media.js
  ├── fetchAPI('articles', {category, limit, offset})  → GAS → 記事管理シート
  ├── fetchAPI('article', {id})                        → GAS → 記事詳細
  ├── fetchAPI('news')                                 → GAS → お知らせシート
  └── fetchAPI('podcasts', {limit})                    → GAS → Podcastシート
```

すべてのコンテンツは **GAS APIを共通ゲートウェイ** として取得する。
既存の `getArticles()`, `getArticleById()`, `getLatestNews()` を活用。

## 2. コラム写真掲載

### 2-1. 画像管理方式

| 項目 | 仕様 |
|------|------|
| **保存先** | さくらサーバー `/site/media/images/{記事ID}/` |
| **アップロードAPI** | `site/api/media/upload_image.php` |
| **対応形式** | JPG, PNG, GIF, WebP |
| **上限** | 1枚あたり5MB、1記事あたり最大10枚 |
| **URL形式** | `https://iba-consulting.jp/site/media/images/{記事ID}/{ファイル名}` |
| **サムネイル** | アップロード時に自動リサイズ（幅600px）のサムネイル版も生成 |

### 2-2. 本文への画像挿入

Markdown本文中に画像URLを記述。メディアサイトで表示時にHTMLに変換。

```markdown
## 相談の様子

![相談風景](/site/media/images/A001/photo1.jpg)

参加者の皆さんに活発な議論をいただきました。
```

### 2-3. GAS管理画面の拡張

記事追加/編集フォームに画像管理UIを追加:

```
[新規記事を追加]
├── 既存項目: タイトル、カテゴリ、タグ、著者、要約、本文、状態、公開日
├── 【新規】サムネイル画像
│   └── [画像をアップロード] → さくらPHP → URL自動入力
└── 【新規】本文用画像
    └── [画像を追加] → アップロード → Markdownタグ生成 → コピーボタン
```

### 2-4. 記事管理シート拡張

| 列 | 項目 | 説明 |
|----|------|------|
| A-L | （既存のまま） | — |
| **M** | ニュース投稿済 | ニュース自動連携フラグ（TRUE/FALSE） |
| **N** | 画像URL一覧 | アップロード済み画像URLの改行区切りリスト |
| **O** | 外部ソースURL | note.comなど元記事のURL |
| **P** | 外部ソース名 | `note.com` 等のプラットフォーム名 |

## 3. Podcast機能

### 3-1. 要件

Podcastの音声ファイルを**さくらサーバーにアップロード**し、メディアサイトで再生できるようにする。
外部Podcast配信サービス（Spotify, Apple Podcast等）へのリンクと**QRコード**も掲載する。

### 3-2. Podcastシート（新規作成）

スプレッドシートに「Podcast」シートを新設:

| 列 | 項目 | 説明 |
|----|------|------|
| A | エピソードID | `P001`, `P002`, ... |
| B | タイトル | エピソード名 |
| C | 説明 | エピソードの説明文 |
| D | 音声ファイルURL | さくらサーバー上のURL |
| E | 再生時間 | `mm:ss` 形式 |
| F | 公開日 | `yyyy.MM.dd` |
| G | 状態 | `draft` / `published` |
| H | Spotify URL | 外部配信リンク |
| I | Apple Podcast URL | 外部配信リンク |
| J | YouTube URL | 外部配信リンク |
| K | サムネイルURL | エピソードのカバー画像 |
| L | 関連記事ID | 記事管理シートの記事IDと紐付け |
| M | 作成日時 | 自動記録 |

### 3-3. 音声ファイルアップロード

| 項目 | 仕様 |
|------|------|
| **保存先** | さくらサーバー `/site/media/podcast/{エピソードID}/` |
| **アップロードAPI** | `site/api/media/upload_podcast.php`（既存 `upload.php` と類似構成） |
| **対応形式** | MP3, M4A, WAV, OGG |
| **上限** | 200MB（90分相当） |
| **URL形式** | `https://iba-consulting.jp/site/media/podcast/P001/episode.mp3` |

### 3-5. QRコード生成

Podcast URLからQRコードを動的生成（表示時にAPIで取得、保存不要）:

```
https://api.qrserver.com/v1/create-qr-code/?size=200x200&data={エンコードURL}
```

代替手段:
- **方式A（推奨）**: 上記外部API（無料・安定・HTTPS対応）
- **方式B**: JavaScriptライブラリ `qrcode.js` でフロントエンド生成

### 3-6. 記事 × Podcast連携

記事管理シートの既存K列（音声ファイルID）を**Podcast エピソードID**に用途変更:

| 列 | 現状 | 変更後 |
|----|------|--------|
| K | 音声ファイルID（未使用） | **Podcast エピソードID**（例: `P001`） |

記事詳細ページで `podcastEpisodeId` が設定されている場合、
Podcastシートからデータを取得してPodcast再生セクションを表示。

## 4. note.com連携（外部コラム自動取込）

### 4-1. 概要

note.comで公開したコラムを**自動的にメディアサイトに取り込む**仕組み。

### 4-2. 技術的な選択肢

| 方式 | 仕組み | メリット | デメリット |
|------|--------|---------|-----------|
| **A: RSSサーバーサイドプロキシ（推奨）** | さくらPHPがnote.comのRSSを取得・パース → 記事管理シートに自動登録 | 安定、CORS問題なし | PHP cron設定が必要 |
| B: 非公式API利用 | note.com内部APIでJSON取得 | データが豊富 | API変更リスクあり（非公式） |
| C: 手動URL登録 | 管理画面でnote.com記事URLを入力 → メタデータ自動取得 | 確実、制御しやすい | 手動作業が発生 |

### 4-3. 推奨: 方式A + 方式C の併用

**定期自動取込（方式A）+ 手動登録の補助（方式C）** を組み合わせる。

#### 自動取込フロー

```
┌──────────────────────────────────────────────────────────┐
│ さくらサーバー (cron: 1日1回)                              │
│                                                          │
│ sync_note.php                                            │
│   1. note.com/{ユーザー名}/rss を取得                      │
│   2. RSS XMLをパース（SimpleXML）                          │
│   3. 既存記事と比較（外部ソースURL = note.com記事URL で重複判定）│
│   4. 新規記事があれば:                                     │
│      a. GAS API に POST（article-add-external アクション）  │
│      b. GAS側で記事管理シートに追加                         │
│         - タイトル: RSSから                                │
│         - 要約: RSS description から                       │
│         - カテゴリ: 「コラム」固定                          │
│         - 本文: 「続きはnote.comでお読みください」+ リンク   │
│         - 外部ソースURL: note.com記事URL（O列）            │
│         - 外部ソース名: 「note.com」（P列）                │
│         - サムネイル: RSS media:thumbnail から              │
│         - 状態: draft（管理者が確認後に published に変更）   │
│   5. ニュースにも自動投稿（公開時）                        │
└──────────────────────────────────────────────────────────┘
```

#### note.com RSSの仕様

| 項目 | 内容 |
|------|------|
| URL | `https://note.com/{ユーザー名}/rss` |
| フォーマット | RSS 2.0 |
| 取得可能項目 | タイトル、要約（抜粋）、URL、公開日、サムネイル画像 |
| **制限事項** | 全文取得は**note proプラン**のみ。無料プランは抜粋のみ |
| CORS | 非対応 → **サーバーサイドでの取得が必須** |

#### 本文の扱い（重要）

note.com無料プランでは全文がRSSに含まれないため、メディアサイト側の表示は:

```
┌─────────────────────────────────────────┐
│ [サムネイル画像]                        │
│                                         │
│ タイトル: 経営課題と診断士の役割         │
│ 著者: ○○  2026.04.01                   │
│                                         │
│ 要約テキスト（RSSのdescriptionから）... │
│                                         │
│ [📎 note.comで続きを読む →]            │
│                                         │
│ ※ この記事はnote.comで公開された        │
│   コンテンツを紹介しています。          │
└─────────────────────────────────────────┘
```

**note proプランの場合**: 全文RSSが利用可能であれば、Markdown変換して本文も表示可能。

#### コメント: note.com連携の判断ポイント

> **note.comをコラム執筆のメインプラットフォームとする場合:**
>
> 1. **note.comで書く → メディアサイトに自動反映** が最も効率的
> 2. 無料プランでも要約+リンクで十分に導線として機能する
> 3. note proプラン（月額8万円〜）にすると全文RSS取得が可能だが、コスト面で要検討
> 4. 将来的にnote.comをやめたい場合も、記事管理シートにメタデータが残るので移行は容易
>
> **メディアサイト側で直接書く場合:**
>
> 1. GAS管理画面の記事投稿フォームから直接Markdown入力
> 2. 写真アップロード機能を使って画像付き記事を作成
> 3. note.com連携は不要（方式Cで必要な時だけ手動取込）
>
> **両方併用（推奨）:**
>
> 1. 長文コラムはnote.comで執筆 → 自動取込でメディアサイトに反映
> 2. 短いお知らせ・イベント報告等はGAS管理画面から直接投稿
> 3. Podcastは常にメディアサイトに直接アップロード

### 4-4. 実装ファイル（note.com連携）

| ファイル | 操作 | 内容 |
|----------|------|------|
| `site/api/media/sync_note.php` | 新規 | note.com RSS取得・パース・GAS連携スクリプト |
| `site/api/media/config.php` | 新規 | note.comユーザー名、APIトークン等の設定 |
| `gas/articles.gs` | 修正 | `article-add-external` アクション追加、O列・P列対応 |
| `gas/main.gs` | 修正 | `article-add-external` エンドポイント追加 |

さくらサーバーのcron設定（手動）:
```
0 6 * * * /usr/local/bin/php /home/ユーザー/www/site/api/media/sync_note.php >> /home/ユーザー/www/site/api/media/logs/sync.log 2>&1
```

## 5. ニュース × コラム自動連携

### 5-1. 自動投稿ルール

| 条件 | 内容 |
|------|------|
| トリガー | 記事管理シートで状態が `published` に変更された時 |
| 対象 | 全カテゴリ |
| ニュース内容 | カテゴリに応じた文面（下記参照） |
| 日付 | 記事の公開日（未設定なら当日） |
| 表示フラグ | `TRUE` |
| 重複防止 | M列（ニュース投稿済）フラグで判定 |

ニュースの文面:
- カテゴリ「コラム」→ `コラム「{タイトル}」を公開しました。`
- カテゴリ「事例紹介」→ `事例紹介「{タイトル}」を公開しました。`

## 6. ステータスフロー改定

### 6-1. 改定内容

```
【現状】
仮予約 → 同意済 → 書類受領 → 確定 → 完了

【改定後】
仮予約 → 同意済 → 書類受領 → 予約確定 → 完了 → 最終完了
                                           ↑          ↑
                                    アンケート受領  報告書送付
```

| ステータス | 値 | タイミング | 変更点 |
|------------|-----|-----------|--------|
| 仮予約 | `仮予約` | 申込受付 | — |
| 同意済 | `同意済` | NDA/同意書に同意 | — |
| 書類受領 | `書類受領` | 事前資料受領 | — |
| **予約確定** | **`予約確定`** | 会場・日程確定 | **旧「確定」から名称変更** |
| **完了** | **`完了`** | **アンケート受領** | **判定条件変更（旧: アンケート送付時）** |
| **最終完了** | **`最終完了`** | **報告書送付** | **新設** |
| キャンセル | `キャンセル` | いずれかの段階 | — |

### 6-2. 完了判定の変更

| 項目 | 現状 | 変更後 |
|------|------|--------|
| 完了の判定 | アンケートを**送付**した時点 | アンケートを**受領**した時点 |
| 判定方法 | アンケートメール送信トリガー | 管理者が手動で変更 or アンケート回答検知（Phase 2） |

**運用フロー:**
```
相談実施 → アンケートメール送信 → 相談者がアンケート回答
                                       ↓
                            スプレッドシートにQ2-Q10記録
                                       ↓
                            管理者確認 → ステータスを「完了」に変更
```

### 6-3. 最終完了の運用フロー

```
完了（アンケート受領済）
  ↓
リーダーが報告書を作成
  ↓
管理者が確認・承認 → 報告書を相談者にメール送信
  ↓
ステータスを「最終完了」に変更
```

**自動化案（Phase 2）**: レポートステータス（Z列）が `配信済` になった時点で自動的に「最終完了」に更新。

## 8. LP連携の段階的アプローチ

### Phase 1: 独立運用（今回実装範囲）

- メディアサイト `site/media/` を構築・公開
- LP・PRサイトからは **ナビゲーションリンクのみ追加**（`<a href="media/">メディア</a>`）
- LP本体には一切手を入れない

### Phase 2: リンク連携

- LP・PRサイトのヘッダーナビに「メディア」メニューを追加
- メディアサイトのヘッダーに「相談申込（LP）」への導線を設置
- 相互に自然な遷移が可能に

### Phase 3: データ連携

- LPのニュースティッカー（既存のお知らせ表示）が、
  メディアサイトと同じGAS API（`getLatestNews()`）からデータを取得
- コラム記事のサムネイル+リンクをLPの「お知らせ」セクションに表示
- **LPの構造を変更せずに、APIレベルで連携**

## 9. 検証ポイント

### メディアサイト基本機能
1. `iba-consulting.jp/site/media/` にアクセスして正常表示
2. お知らせ一覧がGAS APIから取得・表示される
3. コラム一覧がカテゴリフィルタ付きで表示される
4. コラム詳細ページでMarkdown本文がHTMLに変換表示される
5. 既存PRサイトと同じデザインシステム（ヘッダー・フッター・配色）

### 写真
6. GAS管理画面から画像をアップロード → さくらサーバーに保存される
7. コラム詳細ページで画像が表示される
8. サムネイルがコラム一覧のカードに表示される

### Podcast
9. GAS管理画面からPodcastエピソードを登録
10. 音声ファイルをアップロード → さくらサーバーに保存される
11. Podcastページでエピソード一覧・HTML5再生ができる
12. QRコードが正しく表示される
13. 関連コラムとの相互リンクが機能する

### note.com連携
14. `sync_note.php` を実行 → 新着記事が記事管理シートに追加される
15. 外部記事にはnote.comへのリンクと出典表示がある
16. 同じ記事が重複登録されない

### ニュース自動投稿
17. コラムを `published` に変更 → お知らせシートに自動追加
18. メディアサイトのニュース一覧に反映される
19. 重複投稿されない

### ステータスフロー
20. ポータルのフィルターに「予約確定」「最終完了」が表示される
21. 会場確定でステータスが「予約確定」になる
22. 旧「確定」データも正しくフィルタリングされる（後方互換）
23. 管理者が手動で「完了」「最終完了」に変更できる
24. メール通知に「予約確定」が反映されている
